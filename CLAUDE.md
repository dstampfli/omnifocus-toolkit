# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Two OmniFocus automation utilities, a local MCP server that exposes them to Claude Desktop, plus their shared scaffolding:

- `omnifocus_inbox_triage.py` reads every *open* OmniFocus Inbox task and uses the Claude API to classify each one against the user's existing active projects, moving high-confidence matches into their project. Each item's note and any PDF/image attachments are sent to a vision-capable model as part of classification; when `X_BEARER_TOKEN` is set, the text of any linked X (Twitter) post is fetched via the X API and included too.
- `omnifocus_task_reviewer.py` reviews not-yet-reviewed tasks in named OmniFocus project(s), enriching each in place — fetching any linked page via the model's web_fetch (and, when `X_BEARER_TOKEN` is set, the text of any linked X post via the X API) and reading its attachments to produce a clearer title and an appended, timestamped note summary — then marks it done with a tag (default `reviewed`) so re-runs skip it. Reuses `omnifocus_common`'s helpers.
- `omnifocus_sorter.py` reorders the **top-level tasks** of named OmniFocus project(s) by any of the eight keys OmniFocus sorts by natively (`title`, `status`, `added`, `completed`, `due`, `planned`, `defer`, `dropped`) or by a caller-supplied **tag priority order** (`--by tag` with repeated `--tag`, or the `tag_order` argument): each task sorts by the position of its highest-priority tag, matched case-insensitively by leaf name. Makes **no Claude API call** — two `osascript` round-trips regardless of project size. Dry-run by default; `--apply` writes.
- `omnifocus_mcp_server.py` is a local **stdio** MCP server (FastMCP / `mcp[cli]`) exposing five tools — `triage_inbox`, `review_tasks`, `sort_project`, and the read-only `omnifocus_status` and `list_projects` — to a scheduled Claude Cowork task in Claude Desktop. It reuses the toolkit modules directly: `triage_inbox`/`review_tasks` call `run_triage`/`run_review`, `sort_project` calls `run_sort`, and `omnifocus_status`/`list_projects` call `read_omnifocus` — `omnifocus_status` returns just the open-Inbox/active-project counts, while `list_projects` returns each active project's id, name, folderPath, and description so a scheduled agent can discover project names dynamically (e.g. to fan a `review_tasks` call out over every active project). `apply` defaults false; each tool wraps failures into `{"error": ...}`. No network/auth/tunnel — Claude Desktop launches it locally over stdio.
- `omnifocus_common.py` holds task-enrichment helpers (`clean_note`, `media_type_for`, `attachment_block`, `extract_webloc_url`, `build_task_content`, `run_jxa`, `fetch_attachment_b64`, `_positive_int_env`) shared by the triage and reviewer scripts. `build_task_content` turns a `.webloc` attachment into a "Linked web page(s)" text line (URL parsed by `extract_webloc_url`) rather than a vision block, so the model can fetch the link.

- `omnifocus_kanban_plugin/of-kanban-board.omnifocusjs` is a vendored, modified copy of the Omni Automation "Kanban Board" plug-in (v1.2): it adds a `Reviewed` lane (first column, `checkmark.seal` icon) to the upstream `To Do / In Progress / Waiting / Done`. It pairs with `omnifocus_task_reviewer.py`, which tags reviewed tasks `Kanban ▸ Reviewed`. See `omnifocus_kanban_plugin/README.md`.

`main.py` is the unused `uv`-generated placeholder entry point.

## Commands

```bash
python3 omnifocus_inbox_triage.py            # dry-run: classify Inbox tasks and report, change nothing
python3 omnifocus_inbox_triage.py --apply    # classify, then move high-confidence matches into their project

uv sync                                      # install the anthropic/pydantic dependencies
uv run pytest                                # run the unit tests
```

`omnifocus_inbox_triage.py` follows a dry-run-by-default / `--apply`-to-write safety model: dry-run is the implicit default and `--apply` opts into writing.

```bash
python3 omnifocus_task_reviewer.py "Training"          # dry-run: show proposed enrichments
python3 omnifocus_task_reviewer.py "Training" "Tech"   # multiple projects
python3 omnifocus_task_reviewer.py "Training" --apply  # write: rename, append summary, tag reviewed
```

`omnifocus_task_reviewer.py` follows the same dry-run-by-default / `--apply`-to-write model as the triage tool.

```bash
python3 omnifocus_sorter.py "Training" --by due                  # dry-run: show the new order
python3 omnifocus_sorter.py "Training" "Tech" --by title         # multiple projects
python3 omnifocus_sorter.py "Training" --by added --desc --apply # write: newest first
```

`omnifocus_sorter.py` follows the same dry-run-by-default / `--apply`-to-write model.

## Architecture

Both tools embed a JXA (JavaScript for Automation) program run via `osascript -l JavaScript`, with a Python side that handles CLI flags and formats output; all cross-process communication is JSON strings over `osascript`'s argv/stdout.

`omnifocus_inbox_triage.py` inserts a Claude API classification stage between an OmniFocus read and an OmniFocus write, giving it three stages:

1. **Read** (`READ_JXA` / `read_omnifocus`): JXA pulls every *incomplete* Inbox task (completed tasks are skipped — `inboxTasks()` includes finished items) and every project (with id, name, folder path, status, and its OmniFocus note as `description`); the Python side filters to active projects.
2. **Classify** (`classify`): the items and projects are sent to the Claude API via `client.messages.parse(..., output_format=Classification)`, using structured output (Pydantic models `Decision`/`Classification`) so the model returns one typed decision per Inbox item instead of free text to parse. `build_user_message` sends an ordered list of content blocks — a leading text block with the slimmed project taxonomy (id, name, folderPath, `description`), then per item a text header plus a vision block per in-scope attachment (built via `omnifocus_common.build_task_content`). The project note is the primary signal for what belongs where, taking precedence over the project name. Attachment bytes are extracted via `omnifocus_common.fetch_attachment_b64`, which goes through OmniFocus's OmniJS `evaluateJavascript` bridge because plain JXA cannot read attachment contents; the default model is now `claude-sonnet-5`, which must be vision-capable to read them. Classification is batched: `classify_in_batches` splits items via the size-aware `batch_items_by_size` helper, which flushes a batch once it hits `CHUNK_SIZE` items *or* its in-scope attachment bytes would exceed `MAX_BATCH_ATTACHMENT_BYTES`, calls `classify` once per batch, and concatenates the resulting `decisions` lists — keeping each call's output and attachment payload under the model's limits. When `X_BEARER_TOKEN` is set, `build_user_message` also appends any linked X post's text to an item's header via `omnifocus_x` (a run-scoped `XPostFetcher` built in `classify_in_batches` dedupes by post id and caps lookups at `X_FETCH_MAX_USES`, default 25); the fetched text is model input only and never enters the OmniJS write source. With no token the step is skipped.
3. **Apply** (`WRITE_JXA` / `apply_moves`): for decisions that clear `MOVE_MIN_CONFIDENCE`, a second JXA program moves each task into its matched project — run only when `--apply` is passed. The move goes through OmniFocus's **Omni Automation (OmniJS) bridge** (`evaluateJavascript` + `moveTasks(task, project.ending)`), matching tasks/projects by identifier. Note: setting `assignedContainer` from JXA is *not* enough — it only marks a pending assignment and does not relocate the task (yet reports success), which is why the OmniJS bridge is used. Only whitelisted ids (already checked against the real OmniFocus id sets) are embedded into the OmniJS source string — never task names, notes, or other free text.

Config (`MODEL`, `MOVE_MIN_CONFIDENCE`, `CHUNK_SIZE`, plus the attachment/note caps `MAX_ATTACHMENT_BYTES`, `MAX_BATCH_ATTACHMENT_BYTES`, `MAX_NOTE_CHARS`) is read and validated at import by `_load_config()` — a bad `.env` value (non-numeric/non-positive `CHUNK_SIZE` or byte/char cap, an unrecognized `MOVE_MIN_CONFIDENCE`) exits with a clear message instead of a raw traceback, and `MOVE_MIN_CONFIDENCE` is case-normalized. `load_dotenv()` runs first so a gitignored `.env` can supply these plus `ANTHROPIC_API_KEY` (which the `anthropic` SDK reads from the environment directly — no code passes it). `.env.example` is the committed template. All cross-process communication is JSON over `osascript`'s argv/stdout. The pure decision/parsing/prompt-building/reporting logic (everything except the two JXA calls, `fetch_attachment_b64`, and the live API call) is unit-tested under `tests/` — this includes `batch_items_by_size`, but not `classify_in_batches`, which is thin I/O orchestration over the untested `classify`. `omnifocus_common.py`'s pure helpers (`clean_note`, `media_type_for`, `attachment_block`, `build_task_content`) are likewise unit-tested; `run_jxa` and `fetch_attachment_b64` are the I/O boundary and are not. Both `omnifocus_inbox_triage.py` and `omnifocus_task_reviewer.py` now factor their orchestration into a shared `_triage_pipeline` / `_review_pipeline` helper that both the CLI `main()` and a structured-result `run_triage` / `run_review` call — `run_*` returns a JSON-serializable dict consumed by `omnifocus_mcp_server.py`, while `format_report` (and its tests) are unchanged.

`omnifocus_task_reviewer.py` reuses the same embedded-JXA-over-`osascript` pattern for a three-stage pipeline scoped to named projects instead of the Inbox: **read** (`READ_TASKS_JXA`) walks each named project's tasks entirely in OmniJS — skipping completed, dropped, and any task already on the Kanban board (carrying **any** tag in the `KANBAN_TAG` subtree — the `Kanban` parent or any descendant lane, matched by tag identity, so a task keeps being skipped as it progresses `Reviewed → To Do → In Progress → Done`), with a fallback skip on any tag named `REVIEW_TAG` (guards a stray top-level `Reviewed` predating the reparent) — and returns each task's id, name, note, and attachment metadata; **review** (`review_task`) makes one `client.beta.messages.parse(..., output_format=Enrichment)` call per task with the `web_fetch` beta tool (capped at `WEB_FETCH_MAX_USES` uses) enabled, so the model can follow a URL in the note before producing a typed `Enrichment` (`new_title`, `summary`) — failures are caught per task (`review_tasks`) so one bad task never aborts the run; **apply** (`WRITE_JXA` / `apply_enrichments`), run only under `--apply`, sets `name`/`note` **and** adds the tag in a single OmniJS `evaluateJavascript` call. The review tag is resolved as a **child of the `Kanban` parent** (`KANBAN_TAG`, default `Kanban`): the apply stage finds or creates the `Kanban` parent, then finds or creates the `Reviewed` child under it — and reparents any legacy top-level `Reviewed` tag under `Kanban`, so the structure self-heals and reviewed tasks appear in a dedicated `Kanban ▸ Reviewed` board lane. `KANBAN_TAG` is reviewer-specific config alongside `REVIEW_TAG` and `WEB_FETCH_MAX_USES`. The note must be set through OmniJS, not plain JXA: assigning `t.note` via the JXA/Apple-Events bridge replaces the note's rich text and destroys its embedded attachments (`.webloc` links, PDFs, images), whereas the OmniJS assignment preserves them. To keep the model's free text out of the OmniJS *source* (injection safety), each title/note is percent-encoded with `encodeURIComponent` in JXA — whose output cannot break out of a JS string literal — and decoded with `decodeURIComponent` inside OmniJS; only task ids, that encoded text, and the (trusted, config) tag names (`REVIEW_TAG` and `KANBAN_TAG`) reach the source. `MAX_ATTACHMENT_BYTES`/`MAX_NOTE_CHARS` are shared with the triage tool's config; `REVIEW_TAG` and `WEB_FETCH_MAX_USES` are reviewer-specific, validated the same way (`_positive_int_env`, now defined in `omnifocus_common.py` and imported by both `omnifocus_inbox_triage.py` and `omnifocus_task_reviewer.py`). Both `omnifocus_inbox_triage.py` and `omnifocus_task_reviewer.py` now factor their orchestration into a shared `_triage_pipeline` / `_review_pipeline` helper that both the CLI `main()` and a structured-result `run_triage` / `run_review` call — `run_*` returns a JSON-serializable dict consumed by `omnifocus_mcp_server.py`, while `format_report` (and its tests) are unchanged.

`omnifocus_sorter.py` reuses the embedded-JXA-over-`osascript` pattern for a three-stage pipeline, but with **no model stage**: **read** (`READ_SORT_TASKS_JXA` / `read_project_tasks`) walks each named project's **direct children** (`project.children`, *not* `flattenedTasks` — action-group subtasks keep their own order) in OmniJS and returns each task's id, name, status name, and the six date fields as **epoch milliseconds or null** (numeric comparison sidesteps timezone/format ambiguity); unresolvable project names are collected into `missing` rather than aborting. **sort** (`sort_tasks`, pure Python and unit-tested) orders tasks by one of the eight `SORT_KEYS` — tasks with no value for the key always sort **last in both directions** (nulls are split out before sorting, so `descending` can never float them to the top), and the sort is **stable**, so ties keep their input order and re-sorting is idempotent. `status` uses `STATUS_RANK`, an explicit **urgency-first** rank (`Overdue → DueSoon → Next → Available → Blocked → Completed → Dropped`) rather than the meaningless OmniJS `Task.Status.all` declaration order; an unrecognized status sorts after all known ones. An unknown sort key raises `SystemExit` with the valid keys, validated *before* any OmniFocus read. A ninth basis, `tag`, is not a `SORT_KEYS` member (it needs the caller's `tag_order` list) and is validated separately by `_validate_sort`: `by="tag"` with an empty/missing `tag_order` raises `SystemExit` before any OmniFocus read. Its sort value is the minimum `tag_order` index among a task's tags (earliest-listed tag wins), or null — so unmatched tasks fall into the same null bucket that always sorts last. The read stage returns each task's leaf tag names (`tags`) for this; only task/project identifiers still reach the write source, so tag sorting adds no new free text to escape. **apply** (`WRITE_SORT_JXA` / `apply_order`), run only under `--apply`, moves each task to `project.ending` in sorted order — appending in sequence leaves the project in exactly that order. Only project/task **identifiers**, whitelisted by `build_write_config` against the id set the read stage returned, reach the OmniJS source; there is no model free text to percent-encode (unlike the reviewer). `_sort_pipeline` skips the write entirely for a project whose order already matches, and `run_sort` reports per-project `changed` so a scheduled agent can see that. No new env config — `by`/`descending` are per-call arguments — and no per-call batching cap, since sorting makes no API calls and so has no tool-timeout pressure regardless of project size.

When `X_BEARER_TOKEN` is set, `review_task` also appends any linked X post's text to the task content via `omnifocus_x` (a run-scoped `XPostFetcher` built in `review_tasks` dedupes by post id and caps lookups at `X_FETCH_MAX_USES`, default 25, reusing triage's env vars); the model's `web_fetch` tool stays enabled for non-X URLs, the fetched X text is model input only and never enters the OmniJS write source, and with no token the step is skipped.

## Runtime requirements

- macOS with OmniFocus 3/4 installed.
- First run prompts for Automation permissions (System Settings → Privacy & Security → Automation) for the terminal app — required for `osascript` to control OmniFocus.
- `omnifocus_inbox_triage.py` requires the `anthropic` and `python-dotenv` packages (installed via `uv sync`) and an Anthropic API key. Provide the key via a local `.env` (copy `.env.example`), an exported `ANTHROPIC_API_KEY`, or an active `ant auth login` profile.
